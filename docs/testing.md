# Test Pyramid — learn-rs

This document describes the six-level test pyramid for the `learn-rs`
workspace. Each level has different speed, isolation, and coverage
characteristics. Run faster levels first; fall back to slower ones when
you need more confidence.

---

## Level 1 — Unit tests (`cargo test`)

**What**: Pure in-process tests for individual functions and modules.
Each crate's `#[cfg(test)]` module exercises its own logic in isolation.
Mocks are used only at hard I/O boundaries (filesystem, network, child
processes).

**Tooling**: Built-in Rust test harness. No extra installs.

**How to run**:
```bash
cargo test --workspace
```

Or against a single crate:
```bash
cargo test -p learn-acquire
```

**Acceptance criteria**: All tests green. Zero `#[ignore]`-tagged tests
running (the harness skips them by default). 100% of public API paths
exercised by at least one test.

---

## Level 2 — Property tests (`proptest`)

**What**: Randomised inputs generated by `proptest` to verify invariants
that must hold for any valid input — e.g., "chunking preserves the total
token count", "a topic slug never exceeds 40 characters", "serialise then
deserialise returns an equal value".

**Tooling**: `proptest = "1"` in `[dev-dependencies]` of the relevant
crates. Currently present in:
- `learn-chunk` — chunk-size invariants, no empty chunks
- `learn-core` — `Topic::new` slug-length and character-set invariants

Expand to: `learn-acquire` (VTT parser: parse ∘ format round-trip),
`learn-retrieve` (RRF scorer: score always in (0, 1]).

**How to run**:
```bash
cargo test --workspace
```
Proptest tests are ordinary `#[test]` functions; the harness runs them
automatically. Failed cases are shrunk and saved to `.proptest-regressions/`.

**Acceptance criteria**: No failing cases across ≥256 randomly generated
inputs per property (proptest default). Regression files committed when a
new failure is found and fixed.

---

## Level 3 — Integration tests (`tests/` workspace member)

**What**: Cross-crate tests that exercise two or more crates together
against real fixtures. The `learn-integration` crate lives at `tests/` in
the workspace root and is a full workspace member.

**Tooling**: Standard Rust integration test harness (`cargo test`). Fixture
files in `fixtures/` are checked in. Binary fixtures (audio) are generated
on demand by `tests/generate_fixtures.sh`.

**How to run**:
```bash
# Generate audio fixture first (requires ffmpeg):
bash tests/generate_fixtures.sh

# Run all non-ignored integration tests:
cargo test -p learn-integration
```

**Acceptance criteria**: All non-`#[ignore]` integration tests green.
Fixture files must not change between runs (deterministic).

---

## Level 4 — Fuzz tests (`cargo fuzz`)

**What**: Structure-aware fuzzing of the VTT parser. The fuzzer generates
arbitrary byte sequences, feeds them to `learn_acquire::vtt::parse_vtt`,
and reports any panics or sanitizer violations.

**Tooling**: `cargo-fuzz` + Rust nightly toolchain. The harness lives in
`crates/learn-acquire/fuzz/`.

Install:
```bash
rustup toolchain install nightly
cargo install cargo-fuzz
```

**How to run** (from workspace root):
```bash
cd crates/learn-acquire

# Seed the corpus from the fixture:
mkdir -p fuzz/corpus/parse_vtt
cp ../../fixtures/short.vtt fuzz/corpus/parse_vtt/short.vtt

# Run for 60 seconds:
cargo +nightly fuzz run parse_vtt -- -max_total_time=60

# Run indefinitely (Ctrl-C to stop):
cargo +nightly fuzz run parse_vtt -- fuzz/corpus/parse_vtt/
```

See `crates/learn-acquire/fuzz/README.md` for full details.

**Acceptance criteria**: Zero panics or sanitizer violations after a
sustained run (≥10 million executions or 5 minutes, whichever comes
first). Any crasher reduced to a minimal reproducer and filed as a bug.

---

## Level 5 — Mutation tests (`cargo mutants`)

**What**: Mutation testing verifies that your tests actually detect
incorrect logic, not just execute code. `cargo mutants` introduces
controlled bugs (mutations) into the source and checks whether the test
suite catches them.

**Tooling**: `cargo-mutants`.

Install:
```bash
cargo install cargo-mutants
```

**How to run**:
```bash
# Mutation test the chunker and claim-ID derivation in learn-core:
cargo mutants -p learn-chunk -p learn-core
```

Focus targets:
- `learn-chunk`: `chunk_segments`, the token-count accumulator, boundary
  detection logic.
- `learn-core`: `Topic::new` slug derivation, `Manifest` update logic.

**Acceptance criteria**: Mutation score ≥ 75% on targeted modules (i.e.,
at least 3 in 4 mutants are killed by the existing test suite). Surviving
mutants reviewed and either covered by a new test or documented as
"equivalent mutant".

---

## Level 6 — E2E smoke (`#[ignore]`-tagged integration test)

**What**: Full pipeline test that acquires the fixture VTT, chunks it,
embeds with a mock embedder, writes to a temp `.rvf`, queries it, and
asserts that hits are non-empty. This pins the end-to-end API shape before
implementation is complete.

**Tooling**: Same as Level 3. Requires Phase 1 model files (whisper, ort/
BGE weights) to be present; the test is `#[ignore]`-tagged until they are.

**How to run** (after Phase 1):
```bash
cargo test --workspace -- --include-ignored e2e_ingest_and_retrieve_short_fixture
```

Test location: `tests/tests/integration_smoke.rs`

**Acceptance criteria**: Test passes end-to-end. Query against the indexed
fixture returns at least one `Hit` with `score > 0.0`. Runs in under 30
seconds on M-series Mac with models cached.

---

## Quick reference

| Level | Command | Speed | Isolation |
|-------|---------|-------|-----------|
| 1 Unit | `cargo test --workspace` | < 5 s | Full |
| 2 Property | `cargo test --workspace` | < 10 s | Full |
| 3 Integration | `cargo test -p learn-integration` | < 30 s | Fixtures only |
| 4 Fuzz | `cargo +nightly fuzz run parse_vtt` | Minutes–hours | Harness |
| 5 Mutation | `cargo mutants -p learn-chunk -p learn-core` | Minutes | Full |
| 6 E2E smoke | `cargo test -- --include-ignored` | < 30 s | Requires models |
